02 · 五步详解

五步详解

每一步的结构：目标是什么 → 用什么工具 → Jordan 介入点 → 产出 → 如果没这一步会怎样。

1
技术方案输出 // 上游定方向
👤 Jordan 主导
🎯目标
把设计稿（.fig）和产品方案转化成可执行的技术方案：模块划分、目录结构、API 接口定义、状态管理方案
🔧工具
Claude 深度阅读设计稿和产品文档，Jordan 自己输出技术方案文字版
👤Jordan
Jordan 是方案输出主体。Claude 辅助理解设计稿，不是 Jordan 把方案甩给 AI 做
📦产出
文字版技术方案：模块清单 / 目录结构 / 接口字段定义 / 状态设计 / 灰度开关设计
⚠️无此步
Cursor 不知道项目的目录规范、API 约定、组件分层——边写边想，模块边界模糊，后期返工成本高
2
Cursor 分批实现 + 实时 Review // 中游把质量
🤖 Cursor 主力
🎯目标
按技术方案分模块实现，每批实现完 Jordan 立即检查，及时纠正偏差
🔧工具
Cursor Composer — 先搭目录框架，再按模块分批实现，每批完成主动自审
👤Jordan
每批实现完立即 Review：接口字段 / 类型定义 / 异常处理 / 是否符合技术方案；要求 Cursor 主动自审自己的代码，不是等 Jordan 来找 bug
📦产出
已 Review 的模块代码 + 经 Cursor 自审后的版本，每批质量有 Jordan 签字确认
⚠️无此步
等全部写完再返工：bug 定位成本高、上下文已丢失、Cursor 需要重新理解全局，修复效率低
3
全局补漏 + 边缘场景 // 主动预判
🤖 Cursor 补位
🎯目标
所有模块完成后，Cursor 整体检查：对照技术方案查漏、主动分析边缘场景（异常 / 空数据 / 边界值）
🔧工具
Cursor — 整体审查模式，主动列出「还可能出什么问题」
👤Jordan
审查 Cursor 的补漏清单，确认采纳或补充；不是等 bug 出现再修，是主动预判
📦产出
边缘场景覆盖清单 + 已处理的兜底逻辑（空状态 / 超时重试 / 边界值）
⚠️无此步
上线后才发现空数据页面没兜底、库存=0 没做拦截——用户看到白屏或报错，体验差且修起来被动
4
手动调 UI 细节 // 下游做验收
👤 纯手工
🎯目标
对照设计稿手动核验：间距 / 颜色 / 字体大小 / 交互细节，确保 AI 产出与设计稿一致
🔧工具
浏览器 DevTools / Figma 原型 — Jordan 逐项对照设计稿检查
👤Jordan
这是纯人工的工作，AI 不擅长也不应该替代。Jordan 分得清 AI 的边界
📦产出
像素级还原设计稿的 UI 输出，可以交付给 QA 测试的版本
⚠️无此步
AI 生成代码的间距 / 颜色 / 字体与设计稿有偏差，上线后被产品经理或设计师发现——影响专业形象
5
Apifox Mock 测试 + 切换真实接口 // 工程化验证
👤 Jordan 验证
🎯目标
用 Mock 模拟各种接口调用场景，覆盖正常 / 异常 / 灰度开关全场景；切换真实接口时代码零改动
🔧工具
Apifox — 定义接口结构 / Mock 数据 / 场景化测试（正常 / 空数据 / 500 / 超时 / 灰度过滤）
👤Jordan
编写 Mock 场景用例、审查测试覆盖率；切换真实接口时不做代码改动——说明代码干净、接口抽象到位
📦产出
全场景测试通过的接口层代码，切换成本为零
⚠️无此步
前端等后端接口才能开发，进度受阻；上线切真实接口需要改代码，容易出 bug
03 · 功能案例

功能案例：求购模块 + 供应模块

工作流落地在真实业务模块上——不是 Demo，是实际需求。

Case 01
求购模块
多入口（列表页 / 搜索页 / Banner），多字段筛选，灰度开关控制新逻辑曝光比例。
Case 02
供应模块
多入口详情页，库存边界值处理（库存=0 时显示已售罄），新旧逻辑切换。
🚦 灰度开关 — if-else 示意图
服务端控制，不改代码发版，配置即可切换
📦 求购模块请求 → feature_flag.buy_module == "new" ?
✓ 曝光 30% 用户群
→ 新逻辑
✗ 其余 70% 用户
→ 旧逻辑
⚙️ 服务端配置改 100% → 新逻辑全量 → 无需改代码
→ 一键回滚
上线前
小流量验证
异常时
快速回滚
工程价值
改配置 ≠ 发版
🔍 边缘场景主动补位（Cursor 全局补漏阶段覆盖）
🚫 数据为空：列表为空、接口 500、网络超时——每个异常状态都有兜底 UI，不让用户看到白屏
📏 边界值：库存=0 显示「已售罄」、价格超长字段截断、最大输入限制拦截——Jordan 逐项对照设计稿核验
🔀 灰度分支：开关 0% / 30% / 100% 三种情况的前端表现都正确——Apifox Mock 三个场景逐一验证
⏱️ 加载时序：骨架屏 → loading → 内容；error → 重试按钮——完整状态机覆盖，不出现中间态闪烁
04 · 工具链选型

工具链选型

不是随机挑工具，而是「不同工具做它最擅长的事」。

工具 承担角色 选型理由 / 边界
🧠Claude
出方案 / 解决疑难问题
擅长复杂推理和架构判断，能深度理解设计稿和产品文档的意图。
不适合大量重复编码——用它写整个项目是浪费。
⚡Cursor
承担主要编码任务
经济、可实时 Review、分模块实现可控；Cursor 自审机制保证每批产出有基本质量。
生成质量依赖提示词质量，复杂架构决策仍需人判断。
🧪Apifox
Mock 接口 + 场景测试
代码不留调试代码，切换真实接口零改动——接口抽象到位，工程化上线思维。
需要提前定义接口结构，前期有一定定义成本。
核心逻辑一句话：「不同工具做它最擅长的事」 — 如果只用 Cursor 做所有事：架构判断不如 Claude、Mock 能力不如 Apifox，工具边界感清晰才说明有判断力。
