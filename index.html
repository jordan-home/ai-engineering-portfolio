02 · 五步详解

五步详解

每一步 Jordan 和 AI 具体怎么配合，为什么这样分工，哪些必须人做，哪些交给 AI 效率更高。

1
Jordan + Cursor 定技术方案 // 需求到方案的转化
👤 Jordan 主导
🎯目标
把产品方案和设计稿变成可执行的技术方案，不是直接开写
🔧工具
Cursor — Jordan 把产品方案 + 设计稿一起丢给 Cursor，结合前端最佳实践，反复讨论定稿
👤Jordan
Jordan 是方案的输出主体，Cursor 提供前端最佳实践建议，Jordan 判断采纳哪些，最终方案 Jordan 拍板
📦产出
技术方案：模块划分 / 目录结构 / 接口设计 / 状态管理方案 / 灰度开关设计
⚠️无此步
Cursor 不知道项目的目录规范、接口约定、组件分层——边写边想，模块边界模糊，后期返工成本高
2
Cursor 搭框架分批实现 + Jordan 一起 Code Review // 边做边检查
🤖 Cursor 主力
🎯目标
按技术方案分模块实现，每批完成 Jordan 立即检查，及时纠正偏差
🔧工具
Cursor Composer — 先搭目录框架，再按模块一批一批实现，每批完成主动自审
👤Jordan
Cursor 写完一批，Jordan 和 Cursor 一起 code review：接口字段、类型定义、异常处理、是否贴合技术方案——不是等全部写完再返工
📦产出
已 review 的模块代码，每批质量有 Jordan 签字确认
⚠️无此步
等全部写完再返工：bug 定位成本高、上下文已丢失、Cursor 需要重新理解全局，修复效率低
3
全局补漏 // Jordan 推动 Cursor 主动预判
🤖 Cursor 补位
🎯目标
所有模块写完后，Cursor 整体审查一遍，主动列「还可能有什么问题」
🔧工具
Cursor — 整体审查模式，主动列出边缘场景清单（异常、空数据、边界值）
👤Jordan
是 Jordan 推动 Cursor 做的，不是等 bug 出现再修。Jordan 审查补漏清单，确认采纳或补充
📦产出
边缘场景覆盖清单 + 已处理的兜底逻辑（空状态、超时重试、边界值）
⚠️无此步
上线后才发现空数据页面没兜底、库存=0 没做拦截——用户看到白屏或报错，体验差且修起来被动
4
手动调 UI 细节 // 设计稿还原这一步必须人做
👤 纯手工
🎯目标
对照设计稿手动核验：间距、字号、动画效果——AI 生成后这些细节必须人逐项调
🔧工具
浏览器 DevTools / Figma 原型 — Jordan 逐项对照设计稿检查
👤Jordan
AI 不擅长也不应该替代这一步。间距差几个像素、字号大了一号、动画节奏不对——这些必须人眼对照设计稿来调
📦产出
像素级还原设计稿的 UI 输出，可以交付给 QA 测试的版本
⚠️无此步
AI 生成代码的间距 / 颜色 / 字体与设计稿有偏差，上线后被产品经理或设计师发现——影响专业形象
5
Apifox Mock 测试所有业务场景 // Jordan 自己想的边缘场景也要测
👤 Jordan 验证
🎯目标
用 Mock 模拟各种接口调用场景，覆盖正常场景 + 产品需求里写的场景 + Jordan 自己想的边缘场景
🔧工具
Apifox — Mock 各种场景：正常返回、空数据、接口 500、超时、灰度开关不同状态
👤Jordan
不只是测产品需求里的业务场景，Jordan 还会想自己的边缘场景：接口异常了页面怎么表现、返回数据格式不对怎么兜底。切换真实接口时代码零改动
📦产出
全场景测试通过的接口层代码，切换真实接口成本为零——说明代码干净、接口抽象到位
⚠️无此步
前端等后端接口才能开发，进度受阻；上线切真实接口需要改代码，容易出 bug
03 · 功能案例

功能案例：uuskins.com 求购 + 供应模块

工作流落地在真实业务模块上——CS2 饰品交易平台，日均万级订单量。

Case 01
求购模块
买家提交意愿单——填写商品信息、期望价格、数量，由 Jordan 负责前端。多入口（列表页、搜索页、Banner 入口），灰度开关控制新逻辑曝光比例。
Case 02
供应模块
卖家商品满足求购条件时主动供应给买家。当库存=0 时显示已售罄，边界值处理到位。
🚦 灰度开关 — 服务端下发
服务端配置控制，不改代码发版，部分用户看到新逻辑，部分用户看到旧逻辑
📦 求购模块请求 → feature_flag.buy_module == "new" ?
✓ 命中灰度群（部分用户）
→ 新逻辑
✗ 其余用户
→ 旧逻辑
⚙️ 服务端配置改 100% → 新逻辑全量 → 无需改代码
→ 一键回滚
上线前
小流量验证
异常时
快速回滚
工程价值
改配置 ≠ 发版
🔍 边缘场景（Jordan 自己想的边缘场景 + 全局补漏阶段覆盖）
🚫 接口异常：接口 500、网络超时——每个异常状态都有兜底 UI，不让用户看到白屏
📏 边界值：库存=0 显示「已售罄」、价格超长字段截断、最大输入限制拦截——Jordan 逐项对照设计稿核验
🔀 灰度分支：开关 0% / 部分曝光 / 100% 三种情况的前端表现都正确——Apifox Mock 三个场景逐一验证
⏱️ 加载时序：骨架屏 → loading → 内容；error → 重试按钮——完整状态机覆盖，不出现中间态闪烁
04 · 工具链选型

工具链选型

不是挑最贵的工具，是挑最合适的。Jordan 一直在用的真实工具链。

工具 承担角色 选型理由 / 边界
⚡Cursor
一直在用，提效明显
经济、可实时 Review、分模块实现可控；Jordan 和 Cursor 一起讨论定方案、一起 code review，真正有来回的协作。
生成质量依赖提示词质量，复杂架构决策仍需人判断。
🧪Apifox
Mock 接口 + 场景测试
代码里不留调试代码，切换真实接口时代码零改动——说明接口抽象到位，工程化上线思维。
需要提前定义接口结构，前期有一定定义成本。
🧠Claude
适合定方案（本次没用到）
擅长复杂推理和架构判断，适合在需要深度方案讨论的场景里出力。
本场景 Jordan 选择直接在 Cursor 里讨论方案，Claude 本次未使用，但知道它适合哪里。
核心逻辑一句话：「不同工具做它最擅长的事」 — Cursor 一直在用，提效明显；Apifox 让代码干净、切换零成本；Claude 适合定方案，知道边界在哪里才是真懂。
