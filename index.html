02 / Real Cases

实战案例：求购模块 + 供应模块

两个核心业务模块的真实实现——多入口、灰度开关、异常分支全覆盖。

🚦 灰度开关控制
服务端控制曝光比例，按用户群/百分比切换新旧逻辑——不是改代码发版，而是配置开关。
⚙️服务端配置feature_flag: { buy_module: 30% }
→
📦求购模块 (30% 用户)
→
✅新逻辑
⚙️feature_flag 同配置
→
📦供应模块 (多入口)
→
🔄新旧逻辑切换
🧠 工程化思维：上线前通过服务端开关控制曝光，灰度验证后逐步放量，故障可快速回滚
🧪 Apifox Mock 接口测试
用 Apifox 模拟各种接口调用场景，覆盖正常、异常、边界情况，不依赖后端即可验证前端逻辑。
Apifox — 求购模块 Mock 场景 
// 场景 1：正常返回
GET /api/buy/list
→ Response: { items: [...], total: 128, page: 1 }

// 场景 2：空数据
GET /api/buy/list?category=xxx
→ Response: { items: [], total: 0, page: 1 }
// 前端渲染"暂无求购信息"空状态

// 场景 3：网络异常
GET /api/buy/list
→ Response: 500 { error: "内部错误" }
// 前端捕获错误，显示重试按钮

// 场景 4：数据被灰度过滤（开关=30%）
GET /api/buy/list?flag=new_flow
→ Response: { items: [...], flag: "new" }

// 场景 5：供应模块 — 库存边界值
GET /api/supply/detail?id=xxx
→ Response: { stock: 0, status: "sold_out" }
// 前端显示"已售罄"状态
🔍 边缘场景主动补位
🚫 数据为空：列表为空、网络超时、接口 500——每个状态都有兜底 UI
📏 边界值：库存=0、价格超长、最大输入限制——Cursor 实现后 Jordan 逐项对照设计稿核验
🔀 灰度分支：开关 0%/30%/100% 三种情况的前端表现是否都正确处理
⏱️ 加载时序：骨架屏 → loading → 内容；loading → 错误 → 重试——完整状态机覆盖
03 / Engineering Mindset

人机协同的工程思维

不是"AI 厉害我躺平"，而是把 AI 当成强力工具，在关键节点用人的判断力兜底。

💡 核心认知
🧭 人在上游定方向：技术方案输出必须由人来做。Cursor 再强，也不知道你项目的目录规范、API 约定、设计语言——这些只有你最清楚。
🔬 人在中游把质量：Cursor 每批实现完，立即 review。发现偏差当场纠正，而不是等全部写完再返工。
🎨 人在下游做验收：设计稿的间距、颜色、字体——AI 难以 100% 还原。Jordan 手动逐项核验，确保交付质量。
🔄 流程可复盘：每个模块的实现路径都有记录，出现问题可以追溯到哪个环节、谁的决策出了问题。
